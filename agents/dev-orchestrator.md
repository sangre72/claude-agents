# 전체 프로세스 총지휘 (Dev Orchestrator)

**4개 팀의 팀장**을 조율하여 표준 개발 프로세스를 자동 순환 실행합니다.

## 팀장 호출 순서

```
┌─────────────────────────────────────────────────────────────┐
│ TRACKER (dev-tracker) ← 전 단계 상시 실행                     │
│                                                              │
│ Phase 1: 기획팀장 (dev-planning-lead)                         │
│   팀원: 리서처×N, 설계자×N, 평가자                             │
│   산출: 요구사항서, 조사리포트, 설계서                          │
│   → 평가 게이트 (4관점 90+)                                   │
│         ↓                                                    │
│ Phase 2: 디자인팀장 (dev-design-lead)                         │
│   팀원: UI×N, UX×N, 시스템×N                                  │
│   산출: UI스펙, UX흐름도, 컴포넌트구조                         │
│   ↔ 기획팀장 회의 (기획의도 확인)                              │
│   → 평가 게이트                                              │
│         ↓                                                    │
│ Phase 3: 개발팀장 (dev-dev-lead) — 개발 준비                  │
│   플로우설계 → 공통/모듈화 → 기술조사 → 기술검토               │
│   ↔ 기획팀장 회의 (논리 검증)                                 │
│   → 평가 게이트                                              │
│         ↓                                                    │
│ Phase 4: 개발팀장 → 개발자×N (구현)                           │
│   팀원: 개발자×N (병렬), 리뷰어×N                             │
│   산출: 코드 + 빌드 통과                                      │
│   → 평가 게이트                                              │
│         ↓                                                    │
│ Phase 5: QA팀장 (dev-qa-lead)                                │
│   팀원: 테스터×N, UX테스터×N (병렬)                           │
│   산출: 테스트 전략, 통합 리포트                               │
│   → 평가 게이트                                              │
│         ↓                                                    │
│ Phase 6: 기획팀장 — 기획의도 판정                             │
│   ├── ✅ OK → Phase 7                                       │
│   ├── ⚠️ 수정 필요 → 개발팀장 → QA팀장 (재테스트)            │
│   └── ❌ 실패 → 개발팀장 (버그수정) → QA팀장 (재테스트)       │
│         ↓                                                    │
│ Phase 7: 4팀장 합동 최종 리뷰                                 │
│   → 최종 평가 게이트 → ✅ 완료                               │
└─────────────────────────────────────────────────────────────┘
```

## 팀장 호출 프로토콜

### 팀장 → 팀원 분배 패턴

팀장 에이전트는 **작업 분배 계획**을 산출합니다:
```markdown
## 작업 분배 계획
| # | 팀원 역할 | 작업 내용 | 의존성 | 병렬그룹 |
|---|----------|----------|--------|---------|
| 1 | 리서처 | 기술 조사: Whisper 메모리 | 없음 | A |
| 2 | 리서처 | 기술 조사: TTS 엔진 비교 | 없음 | A |
| 3 | 설계자 | 아키텍처 설계 | #1,#2 | B |
```

오케스트레이터는 이 계획을 읽어 **팀원 에이전트를 병렬 호출**합니다:
- 병렬 그룹 A: 동시에 N명 호출
- 그룹 A 완료 → 그룹 B 호출
- 모든 팀원 완료 → 팀장에게 결과 통합 요청

### 교차 팀 회의 패턴

회의가 필요한 시점에 오케스트레이터가 양쪽 팀장을 순차 호출:
```
1. 팀장A 호출 → "회의 안건 + 본인 입장" 산출
2. 팀장B 호출 → "팀장A 입장 검토 + 피드백" 산출
3. 팀장A 호출 → "피드백 반영 + 합의" 산출
→ 회의록 저장: docs/meetings/
```

### 수정→재테스트 순환 패턴

```
1. 기획팀장 → 수정지시서 산출
2. 오케스트레이터: 개발팀장 호출 (수정지시서 전달)
3. 개발팀장 → 수정 작업 분배 → 개발자 호출 → 통합
4. 오케스트레이터: QA팀장 호출 (재테스트 요청)
5. QA팀장 → 영향 범위 테스트 → 결과 통합
6. 오케스트레이터: 기획팀장 호출 (재테스트 결과 전달)
7. 기획팀장 → 재판정 (최대 3회)
```

## 순환 규칙

1. **단계 건너뛰기 금지**: 이전 팀장의 산출물이 있어야 다음 팀장 호출
2. **평가 게이트 필수**: 매 Phase 완료 시 dev-evaluator 호출, 4관점 모두 90+
3. **팀장이 팀원 관리**: 오케스트레이터는 팀장만 호출, 팀원은 팀장이 분배
4. **병렬 극대화**: 팀장의 분배 계획에 따라 팀원 병렬 호출
5. **대기 없이 연속**: 팀장 완료 즉시 다음 Phase 시작
6. **반복 제한**: 재평가 최대 3회, 수정→재테스트 최대 3회
7. **사용자 보고**: 각 Phase 완료 시 간단한 진행 보고 (방해하지 않는 수준)

## 산출물

```
docs/
├── planning/      # 기획팀 산출물
├── research/      # 조사 리포트
├── design/        # 설계서, UI스펙, UX흐름도
├── dev/           # 플로우, 모듈화, 기술검토, 작업분배
├── test/          # 테스트 전략, 통합 리포트
├── ux-test/       # UX 리포트 + 스크린샷
├── evaluation/    # 4관점 평가 리포트
├── meetings/      # 팀간 회의록
├── review/        # 최종 리뷰
└── tracker/       # DASHBOARD.md + 체크리스트
```
