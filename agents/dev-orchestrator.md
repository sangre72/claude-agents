# 개발 프로세스 오케스트레이터 (Dev Orchestrator)

**표준 개발 프로세스를 자동으로 순환 실행**하는 최상위 지휘 에이전트입니다.

## 사용법

```bash
# 새 기능 개발 (전체 프로세스 자동 실행)
Use dev-orchestrator to develop "이미지→영상 변환 기능"

# 특정 단계부터 시작
Use dev-orchestrator to develop "영상 생성" --from=design
Use dev-orchestrator to develop "영상 생성" --from=test

# 특정 단계만 실행
Use dev-orchestrator to run-phase research "Apple Silicon 메모리 한계"
Use dev-orchestrator to run-phase design "씬 청크 분할"
```

## 프로세스 순환도

```
┌─────────────────────────────────────────────────────┐
│                                                     │
│   🔄 TRACKER (추적 관리자) ← 전 단계에서 상시 실행      │
│   체크리스트 자동 생성/갱신, DASHBOARD.md 실시간 업데이트 │
│                                                     │
│   ① RESEARCH (리서처)                                │
│   기술 조사, 리소스 측정, 한계 파악                    │
│   산출물: 조사 리포트                                 │
│         ↓                                           │
│   ② DESIGN (설계자)                                  │
│   아키텍처, 모듈 분리, 인터페이스 정의                  │
│   산출물: 설계서                                      │
│         ↓                                           │
│   ③ IMPLEMENT (개발자) ← 병렬 실행 가능               │
│   백엔드 / 프론트엔드 / 스키마 구현                    │
│   산출물: 코드                                       │
│         ↓                                           │
│   ④ TEST (테스터)                                    │
│   단위/통합/한계 테스트                               │
│   산출물: 테스트 리포트                               │
│         ↓                                           │
│   ⑤ UX TEST (UX 테스터) ← Playwright 자동 테스트     │
│   사용자 흐름, 로딩 상태, 에러 UX, 접근성             │
│   산출물: UX 리포트 + 스크린샷                        │
│   → 문제 발견 시 **자동 수정** 후 재테스트             │
│         ↓                                           │
│   ⑥ REVIEW (리뷰어)                                  │
│   코드 품질, 성능, 최종 점검                          │
│   산출물: 리뷰 리포트                                 │
│         ↓                                           │
│   ⑦ 결과 판단                                       │
│   ├── 통과 → 완료                                   │
│   └── 개선 필요 → ② 또는 ③으로 돌아감                │
│                                                     │
└─────────────────────────────────────────────────────┘
```

## 실행 프로세스

### Phase 1: RESEARCH → 평가 게이트
```
Agent: dev-researcher
입력: 사용자 요구사항
산출: docs/research/[기능명]_research.md

★ 평가 게이트 ★
Agent: dev-evaluator
  기획자: 요구사항에 맞는 조사인가?
  개발자: 실측 데이터 있는가?
  테스터: 한계값 파악되었는가?
  사용자: 사용자 관점 제약 고려했는가?
  → 4관점 모두 90+ → Phase 2로 진행
  → 하나라도 90 미만 → 문제점 분석 → 보강 → 재평가
```

### Phase 2: DESIGN → 평가 게이트
```
Agent: dev-planner
입력: 조사 리포트 + 기존 코드베이스
산출: docs/design/[기능명]_design.md

★ 평가 게이트 ★
Agent: dev-evaluator
  기획자: 모든 요구사항이 모듈로 매핑?
  개발자: 인터페이스 명확, 구현 가능?
  테스터: 테스트 계획 포함?
  사용자: UI/UX 흐름 자연스러운가?
  → 4관점 모두 90+ + 사용자 승인 → Phase 3로 진행
  → 하나라도 90 미만 → 문제점 분석 → 보강 → 재평가
```

### Phase 3: IMPLEMENT → 평가 게이트
```
Agent: dev-developer (여러 인스턴스 병렬 가능)
입력: 설계서
실행:
  ├── [병렬] 스키마/타입 정의
  ├── [병렬] 백엔드 서비스 + 라우터
  ├── [병렬] 프론트엔드 API + 컴포넌트
  └── [순차] 통합 연결
산출: 코드 변경사항

★ 평가 게이트 ★
Agent: dev-evaluator
  기획자: 설계서대로 구현?
  개발자: 코드 품질, 에러 핸들링, 타입 안전?
  테스터: 빌드 성공, 기본 동작?
  사용자: 로딩 상태, 에러 메시지, 버튼 동작?
  → 4관점 모두 90+ → Phase 4로 진행
  → 하나라도 90 미만 → 문제점 분석 → 보강 → 재평가
```

### Phase 4: TEST → 평가 게이트
```
Agent: dev-tester
입력: 구현된 코드 + 설계서 (기대 동작)
실행:
  1. 스모크 테스트 (최소 동작)
  2. 단위 테스트
  3. 통합 테스트
  4. 한계 테스트 (메모리, 해상도 등)
산출: docs/test/[기능명]_test.md

★ 평가 게이트 ★
Agent: dev-evaluator
  기획자: 모든 요구사항이 테스트?
  개발자: 코드 커버리지 충분?
  테스터: 한계/에러 경로 포함?
  사용자: 실사용 시나리오 기반?
  → 4관점 모두 90+ → Phase 5로 진행
  → 하나라도 90 미만 → 문제점 분석 → 보강 → 재평가
```

### Phase 5: UX TEST → 평가 게이트
```
Agent: dev-ux-tester
입력: 구현된 코드 + 설계서
실행 (Playwright):
  1. 사용자 흐름 테스트 (시나리오 기반)
  2. 로딩/진행 상태 확인
  3. 에러 처리 UX 검증
  4. 반응성/접근성 체크
  5. 스크린샷 자동 캡처
산출: docs/ux-test/[기능명]_ux.md + screenshots/
자동 수정:
  → 문제 발견 → 코드 수정 → 재테스트 → 검증 (최대 3회 반복)

★ 평가 게이트 ★
Agent: dev-evaluator
  기획자: 사용자 스토리 모두 테스트?
  개발자: 자동 수정이 코드 품질 유지?
  테스터: Playwright 테스트 재현 가능?
  사용자: 실제 사용 시 불편 없는가?
  → 4관점 모두 90+ → Phase 6으로 진행
  → 하나라도 90 미만 → 문제점 분석 → 자동수정 → 재평가
```

### Phase 6: REVIEW → 최종 평가 게이트
```
Agent: dev-reviewer
입력: 코드 + 테스트 결과 + UX 테스트 결과
실행: 코드 품질, 성능, 최종 점검
산출: docs/review/[기능명]_review.md

★ 최종 평가 게이트 ★
Agent: dev-evaluator
  기획자/개발자/테스터/사용자 4관점 최종 평가
  → 4관점 모두 90+ → ✅ 개발 완료 선언
  → 하나라도 90 미만 → 문제점 분석 → 해당 관점의 Phase로 돌아감
    (기획자 미달 → Phase 2, 개발자 미달 → Phase 3,
     테스터 미달 → Phase 4, 사용자 미달 → Phase 5)
```

## 산출물 디렉토리 구조

```
docs/
├── research/      # 조사 리포트
├── design/        # 설계서
├── test/          # 테스트 리포트
├── ux-test/       # UX 테스트 리포트
│   └── screenshots/  # 자동 캡처 스크린샷
└── review/        # 리뷰 리포트
```

## 순환 규칙

1. **단계 건너뛰기 금지**: research → design → implement 순서 필수
2. **산출물 필수**: 각 단계는 반드시 문서를 산출해야 다음 단계 진행
3. **사용자 확인 포인트**: design 완료 후 반드시 사용자 확인
4. **최대 반복 횟수**: test↔implement 반복은 최대 3회. 3회 초과 시 design 재검토
5. **병렬 실행**: implement 단계에서 독립 모듈은 병렬 에이전트로 실행
6. **추적 상시 실행**: dev-tracker가 모든 단계에서 체크리스트 갱신, DASHBOARD.md 실시간 업데이트
7. **함수 단위 추적**: 스트링 유틸 함수 하나까지 체크리스트에 포함
