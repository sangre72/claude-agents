# 표준 개발 프로세스 스킬 (팀 조직 체계)

사용자가 `/dev` 명령어를 실행하면 **4개 팀(기획/디자인/개발/QA)의 팀장-팀원 체계**로 표준 개발 프로세스를 자동 순환 실행합니다.

---

## ⚠️ 최우선 원칙: 동작하는 소프트웨어 > 문서

```
┌──────────────────────────────────────────────────────────────┐
│  "빌드 성공 ≠ 동작 성공. 문서 100장 < 실제 동작 1번."          │
│                                                               │
│  모든 Phase의 완료 기준은 "문서 작성"이 아니라                    │
│  "실제로 브라우저/서버에서 동작하는 것을 눈으로 확인"이다.        │
│                                                               │
│  Playwright 테스트가 통과하지 않으면 어떤 Phase도 완료할 수 없다. │
│  평가 점수 90+이어도 실제 동작이 안 되면 0점이다.                │
└──────────────────────────────────────────────────────────────┘
```

### 동작 검증 게이트 (모든 Phase 전환 시 필수)

| 검증 항목 | 방법 | 실패 시 |
|----------|------|--------|
| 페이지 로드 | Playwright: 4000 접속 → 주요 UI 렌더링 확인 | Phase 전환 차단 |
| 핵심 기능 동작 | Playwright: 변경 기능 + 연계 기능 실제 클릭/입력 | Phase 전환 차단 |
| 콘솔 에러 0건 | Playwright: `page.on('pageerror')` 수집 | P0 즉시 수정 |
| API 호출 성공 | Playwright: `page.on('response')` 상태코드 확인 | P0 즉시 수정 |
| 화면 깨짐 없음 | Playwright: 스크린샷 캡처 → 시각 확인 | 수정 후 재검증 |

### Playwright 필수 테스트 시나리오 (웹 UI 작업 시)

```
1. [직접 기능] 변경한 기능이 실제로 동작하는가?
   - 버튼 클릭 → 예상 결과 확인
   - 입력 → 출력 확인
   - 상태 변경 → UI 반영 확인

2. [연계 기능] 다른 탭/컴포넌트로 데이터가 올바르게 전달되는가?
   - 탭A에서 작업 → 탭B에서 결과 확인
   - 프로젝트 저장 → 불러오기 → 모든 탭 데이터 복원 확인

3. [사용자 시나리오] 실제 사용자가 하는 순서대로 동작하는가?
   - 처음부터 끝까지 전체 흐름 1회 실행
   - 중간에 탭 전환, 뒤로 가기, 다시 진행 등

4. [에러 시나리오] 비정상 상황에서 앱이 깨지지 않는가?
   - 빈 데이터, 네트워크 실패, 잘못된 입력
```

### 평가 점수 산정 기준 변경

```
[기존 — 잘못된 방식]
"문서가 잘 작성되었는가?" → 90+점 → 통과

[변경 — 올바른 방식]
"실제로 동작하는가?" → Playwright 테스트 PASS → 그 위에 문서 품질 가산

실제 동작 검증 없이 90+점을 줄 수 없다.
Playwright 테스트 실패 = 자동 0점 = Phase 전환 차단.
```

## 사용법

```bash
/dev 이미지→영상 변환 기능 추가        # 전체 프로세스 실행
/dev planning 대본 작성 기능 개선       # 기획팀만 실행
/dev design 시나리오 UI 개선            # 디자인팀만 실행
/dev research 영상 생성 메모리 한계     # 기획팀 리서처 실행
/dev implement 이미지→영상 API          # 개발팀만 실행
/dev test 영상 생성 API                 # QA팀만 실행
/dev ux-test 시나리오 영상 생성         # QA팀 UX테스트만 실행
/dev review 시나리오 영상 생성          # 개발팀 코드리뷰 실행
/dev status                             # 전체 진행 현황
/dev checklist 영상 생성                # 세부 체크리스트
```

## 팀 간 커뮤니케이션 프로토콜

```
         기획팀장
        ↗    ↕    ↘
  디자인팀장 ←→ 개발팀장 ←→ QA팀장
        ↘    ↕    ↗
         기획팀장

핵심 흐름:
기획팀장 → 디자인팀장: 기획의도, 사용자 스토리
기획팀장 ↔ 개발팀장: 논리 검증 회의 (플로우가 맞는가?)
QA팀장 → 기획팀장: 테스트 결과 보고
기획팀장 → 개발팀장: 기획의도 불일치 수정 지시
개발팀장 → QA팀장: 수정 완료, 재테스트 요청
```

## Gitignore 규칙 (프로젝트 초기화 시 자동 적용)

`/dev` 명령 최초 실행 시 `.gitignore`에 다음 패턴이 없으면 자동 추가:

```gitignore
# === Dev Process Outputs (auto-managed) ===

# 테스트 산출물 (실행마다 재생성)
test-results/
playwright-report/
blob-report/
tests/screenshots/

# docs/ 디렉토리는 Git에 포함 (산출물 이력 추적)
# 단, 아래는 제외:
docs/ux-test/screenshots/    # UX 테스트 스크린샷 (대용량)
docs/tracker/*.tmp            # 트래커 임시 파일
```

### 파일명 규칙

```

예시:
docs/planning/image_to_video_requirements.md
docs/research/image_to_video_user_research.md
docs/research/image_to_video_tech_research.md
docs/design/image_to_video_design.md
docs/dev/image_to_video_flow.md
docs/dev/image_to_video_modules.md
docs/test/image_to_video_strategy.md
docs/test/image_to_video_report.md
docs/evaluation/image_to_video_phase1_eval.md
docs/meetings/planning-design_image_to_video.md
docs/tracker/DASHBOARD.md
```

### 기능명 규칙
- 영문 snake_case 사용 (한글 파일명 금지)
- 간결하게 핵심 키워드만: `image_to_video`, `character_sheet`, `tts_improvement`
- 동일 기능의 모든 산출물에 같은 기능명 사용

### 산출물 생명주기

```
생성 → 업데이트(수정→재평가 시) → 아카이브(완료 후)

수정 시: 기존 파일 덮어쓰기 (Git 이력으로 버전 추적)
완료 후: docs/tracker/DASHBOARD.md에 완료 표시
```

| Phase | 필수 산출물 | 생성 주체 |

## 전체 프로세스 실행 순서

### Phase -1: 이전 작업 확인 (모든 /dev 실행 시 최우선)

```
⚡ /dev 명령 수신 → 먼저 기존 작업 상태 확인

-1-1. docs/tracker/DASHBOARD.md 존재 여부 확인
      ├── 존재 → 읽고 현재 Phase/상태 파악
      │   ├── 미완료 작업 있음 → 사용자에게 보고:
      │   │   "이전 작업 [{기능명}] Phase {N}에서 중단됨. 이어서 진행할까요?"
      │   │   ├── 사용자 동의 → 해당 Phase부터 재개
      │   │   └── 새 작업 요청 → 기존 DASHBOARD 아카이브 후 새로 시작
      │   └── 모두 완료 → 새 작업으로 진행
      └── 미존재 → docs/ 디렉토리 전체 스캔
          ├── 산출물 존재 → 파일명에서 기능명/Phase 추론, 상태 보고
          └── 산출물 미존재 → 완전 새 시작

-1-2. .gitignore 확인
      - Dev Process 관련 패턴 누락 시 자동 추가
      - 이미 Git 추적 중인 파일이 있으면 `git rm --cached` 처리

-1-3. docs/ 디렉토리 구조 확인/생성
      - 필요한 하위 디렉토리가 없으면 자동 생성
```

### Phase 0: 사용자/시장 리서치 (모든 것에 앞서 실행)

```
⚡ 기능 요청 수신 → 기획 전에 반드시 사용자 리서치 먼저

0-1. 기획팀장 → 사용자 리서처(dev-user-researcher): 웹 조사 지시
     - WebSearch/WebFetch로 실제 웹 전체를 검색:
       사이트, 논문, 블로그, 포럼, GitHub, 경쟁사, YouTube 등
     - 조사 항목:
       a) 일반 기능인가 특화 기능인가? (판별 기준 + 근거)
       b) 사용자가 실제로 원하는 것? (표면 vs 진짜 니즈)
       c) 경쟁/유사 서비스 분석 (기능, UX, 장단점)
       d) 검증된 UX 패턴 (Best Practice 레퍼런스)
       e) 차별화 기회 (우리만의 가치)
     → 산출물: docs/research/[기능명]_user_research.md

0-2. (병렬) 기획팀장 → 기술 리서처(dev-researcher): 최신 안정 기술 조사
     - WebSearch로 최신 stable 릴리스 확인
     - 최신 + 안정 기준: 6개월 이내 업데이트 + SemVer 1.0+ + 프로덕션 사례
     - 최소 2~3개 대안 기술 비교
     → 산출물: docs/research/[기능명]_tech_research.md

0-3. 기획팀장: 리서치 결과 종합 판단
     - 일반 기능 → Best Practice 따르기, 차별화 포인트 1~2개
     - 특화 기능 → 심층 사용자 조사, 프로토타입 검증 계획
     - 사용자 편의성 관점에서 기능 스코프 결정
     → 산출물: docs/planning/[기능명]_research_summary.md

     → Phase 1로 진행 (평가 게이트 선택적)
```

### Phase 0.5: 기능 의존성 그래프 분석 ★★★ 가장 중요

```
⚡ 기능 요청을 받으면 코딩/기획 전에 반드시 전체 데이터 플로우를 추론합니다.
   "영상 음성 머징"이라는 요청 하나에서 아래 모든 것을 자동으로 도출해야 합니다.
   사용자가 하나하나 지시해야 하는 것은 프로세스 실패입니다.

┌──────────────────────────────────────────────────────────────────┐
│  기능 요청 → 의존성 그래프 자동 생성 → 누락 0건 확인 → 기획 시작  │
│                                                                   │
│  "버튼만 만들고 뒤의 데이터 플로우를 빠뜨리면 실패"                 │
│  "API만 만들고 프론트 호출/저장/복원을 빠뜨리면 실패"               │
│  "UI만 만들고 물리적 파일 경로/변환을 빠뜨리면 실패"                │
└──────────────────────────────────────────────────────────────────┘

0.5-0. ★ 기존 코드베이스 검색 (재사용 우선, 중복 구현 금지)
     ┌─────────────────────────────────────────────────────────┐
     │ 새 코드를 작성하기 전에 반드시 기존 코드를 검색합니다.      │
     │ 이미 있는 것을 새로 만드는 것은 "중복 구현"이며 반려됩니다.  │
     │                                                          │
     │ 검색 방법:                                                │
     │ a) grep -r "관련키워드" src/ python-server/               │
     │    - 함수명, 파일경로 패턴, API 엔드포인트, 변수명          │
     │    - 예: "resolveVideoUrl", "getTTSDownloadUrl",          │
     │      "cache/tts", "/tts/download" 등                     │
     │                                                          │
     │ b) 발견한 기존 코드 목록 작성                               │
     │    - 파일 경로 + 함수명 + 역할                             │
     │    - 재사용 가능 여부 판단                                  │
     │    - 확장 필요 시 기존 함수에 파라미터 추가 vs 새 함수       │
     │                                                          │
     │ c) 재사용 결정                                             │
     │    - 기존 유틸/훅/API 있음 → 반드시 재사용                  │
     │    - 기능이 부족 → 기존 것을 확장 (새로 만들지 않음)         │
     │    - 완전히 새로운 기능 → 그때만 새로 생성                   │
     │                                                          │
     │ 같은 로직이 2곳 이상 → 즉시 공통 유틸로 추출               │
     └─────────────────────────────────────────────────────────┘

0.5-1. 입력 데이터 추적 (물리적 위치)
     ┌─────────────────────────────────────────────────────────┐
     │ 모든 입력 데이터의 물리적 위치를 추적합니다:               │
     │                                                          │
     │ a) 파일은 어디에 있는가?                                  │
     │    - 프로젝트 폴더? (projects/{id}/)                     │
     │    - 캐시 폴더? (cache/tts/, cache/qwen_tts/, cache/voices/)│
     │    - 임시 폴더? 외부 URL?                                 │
     │                                                          │
     │ b) URL↔로컬 경로 변환이 필요한가?                         │
     │    - 서빙 URL (/tts/download/file.wav) → 로컬 (cache/tts/file.wav) │
     │    - 프론트에서 보는 URL vs 백엔드에서 접근하는 경로        │
     │                                                          │
     │ c) 파일 형식/코덱은?                                      │
     │    - 영상: mp4(h264), webm(vp9) 등                       │
     │    - 오디오: wav, mp3, aac, ogg 등                       │
     │    - 변환이 필요한가? (FFmpeg 파이프라인)                   │
     └─────────────────────────────────────────────────────────┘

0.5-2. 처리 파이프라인 설계
     - 입력 A + 입력 B → 어떤 처리 → 출력 C
     - 처리 도구: FFmpeg? Web Audio API? 서버 프로세스?
     - 처리 시간: 즉시? 비동기? 폴링 필요?
     - 에러 시나리오: 파일 없음, 코덱 불일치, 타임아웃, 디스크 공간

0.5-3. 출력/저장 설계
     - 결과 파일은 어디에 저장?
     - 파일명 규칙은? (scene_XX_merged.mp4 등)
     - 기존 파일과 충돌 시? (덮어쓰기? 버전?)
     - 프로젝트 JSON에 편집 정보 저장 필요?

0.5-4. 프론트엔드 상태 연계
     - 어떤 React state에 반영?
     - 어떤 컴포넌트가 결과를 표시?
     - 다른 탭/컴포넌트로 데이터가 전달되는가?
     - props 체인: 어디서 어디까지?

0.5-5. 저장/복원 연계
     - 프로젝트 저장 시 이 편집 결과가 포함되는가?
     - getState()에 포함? project.json의 어느 키?
     - 불러오기 시 복원되는가? setState/restoreState에 반영?

0.5-6. API 설계 (백엔드 필요 시)
     - 엔드포인트: GET/POST, 경로, 요청/응답 스키마
     - 요청 파라미터: 필수/선택, 기본값
     - 응답: 성공/실패 형식, 반환 데이터
     - 보안: 경로 순회 방지, 파일 크기 제한, 타임아웃

0.5-7. 의존성 그래프 최종 검증
     ┌─────────────────────────────────────────────────────────┐
     │ 체크리스트 (하나라도 빠지면 기획 시작 금지):               │
     │                                                          │
     │ □ 입력 데이터의 물리적 경로를 모두 파악했는가?              │
     │ □ URL→로컬 경로 변환 로직이 있는가?                       │
     │ □ 처리 결과 파일의 저장 위치/이름이 정해졌는가?             │
     │ □ 프론트 state 반영 경로가 명확한가?                      │
     │ □ 프로젝트 저장/복원에 포함되는가?                         │
     │ □ 에러 시나리오별 처리가 정해졌는가?                       │
     │ □ 연계 기능(다른 탭, 프로덕션)에 영향이 있는가?             │
     └─────────────────────────────────────────────────────────┘

     → 산출물: 의존성 그래프 (기획서 내 포함 또는 별도 문서)
     → 이 분석이 완료되어야 Phase 1 기획 시작 가능
```

### 예시: "영상 음성 머징" 의존성 그래프

```
요청: "영상에 TTS 음성을 합성하는 기능"

[입력 데이터]
├── 영상: projects/{id}/scene_XX.mp4 (h264/aac, 480x848, 5~8초)
├── TTS 음성: 3가지 출처
│   ├── 1) 프로젝트 폴더: projects/{id}/tts_scene_XX.wav
│   ├── 2) Edge TTS 캐시: cache/tts/{filename}.mp3
│   └── 3) Qwen TTS 캐시: cache/qwen_tts/{filename}.wav
└── URL 변환 필요:
    ├── /tts/download/{file} → cache/tts/{file}
    ├── /qwen-tts/download/{file} → cache/qwen_tts/{file}
    └── /project/media/{id}/{file} → projects/{id}/{file}

[처리]
├── FFmpeg: -map 0:v -map 1:a -c:v copy -c:a aac -shortest
├── 원본 오디오 처리: 제거(-an) 또는 믹스(amix)
├── 볼륨 조절: volume 필터 (0.0~2.0)
└── 타임아웃: 180초

[출력]
├── 파일: projects/{id}/scene_XX_merged.mp4
├── 원본 유지: scene_XX.mp4 그대로
└── URL: /project/media/{id}/scene_XX_merged.mp4

[프론트 상태]
├── editedVideoUrl state에 URL 저장
├── showOriginal 토글로 원본/편집본 전환
├── displayVideoUrl 계산식: editedVideoUrl || resolveVideoUrl(rawUrl)
└── videoMeta 업데이트 (편집본 메타데이터)

[저장/복원]
├── ScenarioGeneratorState에 editedVideoUrls 필드 추가?
├── project.json → media.editedVideos?
└── 불러오기 시 편집본 파일 존재 확인 → 자동 복원

[API]
├── POST /project/{id}/merge-audio/{scene}
│   body: { audio_url: string, volume?: number, keep_original_audio?: boolean }
│   응답: { success, video_url, filename }
├── 경로 순회 방지: resolve() + startswith()
└── 에러: 파일 없음(404), 코덱 오류(500), 타임아웃(504)

[연계]
├── 프로덕션 탭: loadFromProject에서 editedVideo 우선 로드?
├── 프로젝트 저장: videoFiles에 편집본 경로 포함?
└── 영상 정보 칩: 편집본의 해상도/길이 표시
```

```
     │                                                          │
     │ [영향 범위 확인]                                           │
     │ - 변경할 파일을 import하는 모든 파일 목록 확인               │
     │ - 타입/인터페이스 변경 시 하위 호환성 확인                   │
     │ - API 시그니처 변경 시 프론트↔백 양쪽 수정 필요 여부 확인    │
     └─────────────────────────────────────────────────────────┘

     - 완료 후 개발팀장에게 보고

4-3. 개발팀장/리뷰어: 코드 리뷰
     ┌─────────────────────────────────────────────────────────┐
     │ 리뷰어 필수 점검 항목:                                     │
     │                                                          │
     │ [표준 준수]                                               │
     │ - 코딩 표준 문서와 일치하는가?                               │

     │ - 개발자에게 수정 지시 → 수정 후 재리뷰                     │
     │ - 최대 2회 반복 후에도 미달 → 개발팀장 에스컬레이션          │
     └─────────────────────────────────────────────────────────┘

     - 패턴 일관성, 에러 핸들링, 타입 안전성
     - 모듈 간 인터페이스 일치 확인
     - 보안 규칙 준수 (위험 함수 사용 여부)

4-4. 공통 개발자: 통합 후 공통 모듈 갱신
     - 리뷰 중 발견된 공통화 가능 패턴 → 공통 모듈로 승격
     - 공통 모듈 문서 업데이트
     - 다음 Phase에서 참조할 수 있도록 최신화

4-5. 개발팀장: 통합 + 빌드 확인
     - 모듈 연결, import 정리
     - 빌드 통과 확인
     - 백엔드 서버 재시작 (자동)

4-6. 개발팀장: 개발자 동작 검증 (Developer Smoke Test) ★★★ 최우선 필수
     ┌─────────────────────────────────────────────────────────┐
     │ 코드 변경 후 반드시 실제 동작을 검증해야 합니다.            │
     │ 빌드 성공 ≠ 동작 성공. 빌드만 확인하고 넘어가면 안 됩니다.  │
     │                                                          │
     │ ★ Playwright로 실제 브라우저에서 동작을 확인해야만          │
     │   Phase 5로 진행할 수 있습니다. 예외 없음.                 │
     └─────────────────────────────────────────────────────────┘

     a) 서버 재시작 후 동작 확인
        - 프론트엔드: pnpm dev 재시작 → 페이지 로드 확인
        - 백엔드: uvicorn 재시작 → /health 엔드포인트 확인
        - 변경된 API 라우트가 있으면 curl로 실제 호출 테스트

     b) ★ Playwright 실제 동작 검증 (필수 — 이것 없이 Phase 전환 불가)
        ┌──────────────────────────────────────────────────────┐
        │ 구현한 모든 기능을 Playwright로 실제 클릭/입력하여      │
        │ 동작하는 것을 증명해야 합니다.                          │
        │                                                       │
        │ "버튼이 있다" ≠ "버튼이 동작한다"                       │
        │ "컴포넌트가 렌더링된다" ≠ "사용자가 쓸 수 있다"          │
        │                                                       │
        │ 핵심 사용자 시나리오를 처음부터 끝까지 실행하여:          │
        │ 1. 기대 결과가 나오는지 (데이터, UI 상태)               │
        │ 2. 콘솔 에러가 0건인지                                 │
        │ 3. 연계 기능(다른 탭, 저장/불러오기)도 정상인지          │
        │ 4. 스크린샷을 증거로 저장                               │
        └──────────────────────────────────────────────────────┘

        Playwright 테스트 작성 필수 포함:
        - page.on('console') → 에러 수집
        - page.on('pageerror') → 런타임 에러 수집
        - page.on('requestfailed') → 네트워크 실패 수집
        - page.screenshot() → 각 단계 스크린샷 저장
        - 실제 버튼 클릭, 입력, 탭 전환 등 사용자 행동 시뮬레이션

     c) 변경 범위별 추가 검증
        - API 엔드포인트 변경 → curl로 정상/에러 응답 확인
        - 보안 수정 → 수정 전 공격 벡터로 재시도하여 차단 확인
        - 프론트 컴포넌트 변경 → Playwright로 실제 UI 동작 확인
        - 환경변수 변경 → 서버 재시작 후 새 값 반영 확인

     d) 미사용 코드 감지
        - 변경/삭제한 파일이 어디서 import되는지 확인 (grep)
        - import 없는 파일 → 삭제 대상으로 표시

     e) 단위 테스트 실행
        - pnpm test (프론트엔드)
        - cd python-server && uv run pytest (백엔드)

     f) 검증 실패 시
        - 즉시 수정 → Playwright 재검증 (최대 3회)
        - 3회 실패 시 접근 방식 재검토 (다른 해결책 모색)

4-7. 기획팀장 → 평가자: 구현 4관점 평가
     ★ 평가자는 반드시 Playwright 테스트 결과(스크린샷 + 로그)를 확인한 후 점수 산정
     ★ Playwright 테스트 없이 제출된 Phase는 자동 0점 처리
     → 4관점 모두 90+ + Playwright PASS → Phase 5로 진행
```

### Phase 5: 테스트 (QA Lead → Testers)

```
5-1. QA팀장: 기획서 기반 테스트 전략 수립
     - 요구사항 ↔ 테스트 케이스 매핑
     - 단계별 실행 계획
     - 산출물: docs/test/[기능명]_strategy.md

5-2. QA팀장: 팀원에게 작업 분배 (대규모 병렬)
     - 테스터A~C: 단위 테스트 (모듈별 분배)
     - 테스터D: 통합 테스트 (전체 흐름 E2E)
     - 테스터E: 한계/동시성 테스트
     - 보안테스터A~B: OWASP 10대 취약점, API 키 노출, 인젝션, XSS 전문 테스트 (dev-security-tester)
     - UX테스터A~C: Playwright 테스트 (화면별 분배)

5-3. 테스트 단계적 실행
     Phase 5a: 스모크 테스트 (서버 실행 상태에서 필수)
       ┌────────────────────────────────────────────────────────┐
       │ 스모크 테스트는 실제 서버를 기동한 상태에서 수행합니다.    │
       │ 빌드 통과 여부가 아닌 실제 HTTP 요청/응답을 검증합니다.   │
       └────────────────────────────────────────────────────────┘
       - 프론트엔드 서버 기동 확인 (http://localhost:4000 접근)
       - 백엔드 서버 기동 확인 (http://localhost:15000/health)
       - 변경된 API 엔드포인트 curl 호출 (정상 요청 + 에러 요청)
       - 보안 수정 항목은 공격 시뮬레이션으로 차단 확인
       - 스모크 실패 시 Phase 4로 회귀 (개발팀 수정 → 재스모크)

     Phase 5b: 단위 테스트 (pnpm test + pytest)
     Phase 5c: 통합 테스트 (E2E 흐름)
     → Phase 5d: 보안 (보안테스터 10개 카테고리 스캔 + 자동수정)
     → Phase 5e: 한계/동시성
     → Phase 5f: UX 동작 테스트 (Playwright + 실제 UI 검증) ★★★ 최종 게이트
       ┌────────────────────────────────────────────────────────┐
       │ UX 테스트는 실제 브라우저에서 사용자 시나리오를 수행합니다. │
       │ 이 테스트를 통과하지 못하면 Phase 6으로 갈 수 없습니다.    │
       │                                                         │
       │ "코드가 있다" ≠ "기능이 된다"                              │
       │ "빌드 통과" ≠ "사용자가 쓸 수 있다"                        │
       │ "단위 테스트 PASS" ≠ "실제 화면에서 동작한다"              │
       └────────────────────────────────────────────────────────┘

       필수 테스트 시나리오:
       a) [핵심 흐름] 변경 기능의 처음→끝 전체 사용자 시나리오
          - 실제 버튼 클릭, 실제 입력, 실제 결과 확인
          - "재생" 버튼 → 실제로 영상이 재생되는가?
          - "드래그" → 실제로 순서가 바뀌는가?
          - "저장" → 실제로 데이터가 저장되고 불러와지는가?

       b) [연계 흐름] 데이터가 다른 탭/컴포넌트로 올바르게 전달
          - 시나리오 탭 → 프로덕션 탭 데이터 전달
          - 프로젝트 저장 → 불러오기 → 모든 탭 복원

       c) [에러 수집] 콘솔/네트워크 에러 0건 확인
          - page.on('pageerror') — 런타임 크래시
          - page.on('console', type=error) — 에러 로그
          - page.on('requestfailed') — 네트워크 실패
          - 기능 관련 에러가 1건이라도 있으면 FAIL

       d) [스크린샷 증거] 모든 주요 단계에서 캡처
          - tests/screenshots/ 에 저장
          - 스크린샷이 없는 테스트는 증거 불충분으로 FAIL

       e) [화면 리프레시 검증] 의도치 않은 리렌더링 확인
          - 특정 동작 후 전체 페이지가 리프레시되지 않는지
          - state 변경이 불필요한 리렌더링을 유발하지 않는지

       자동 수정 가능한 UI 버그는 즉시 수정 → 재테스트

5-4. QA팀장: 결과 통합 → 통합 리포트 작성
     - 산출물: docs/test/[기능명]_report.md

5-5. 기획팀장 → 평가자: 테스트 4관점 평가
     → 4관점 모두 90+ → Phase 6로 진행
```

### Phase 6: 기획 검토 순환 (핵심)

```
6-1. QA팀장 → 기획팀장: 테스트 결과 보고

6-2. 기획팀장: 기획의도 부합 판정
     ├── ✅ 기능 OK + 기획의도 OK → Phase 7로 진행
     ├── ⚠️ 기능 OK + 기획의도 MISMATCH
     │   → 수정지시서 작성
     │   → 개발팀장에게 전달
     │   → 개발팀장: 팀원에게 수정 지시 → 수정 → 리뷰 → 통합
     │   → 개발팀장 → QA팀장: "재테스트 요청"
     │   → QA팀장: 영향 범위 재테스트 + 회귀 테스트
     │   → QA팀장 → 기획팀장: 재테스트 결과 보고
     │   → 기획팀장: 재판정 (최대 3회 반복)
     ├── ❌ 기능 FAIL
     │   → 개발팀장에게 버그 리포트 → 수정 → 재테스트 순환
     └── 🔄 기획 자체 수정 필요
         → 요구사항 변경 → 전 팀에 공유 → Phase 1 부분 재실행
```

## 평가 게이트 (전 단계 공통)

매 단계 완료 시 **기획팀 소속 dev-evaluator**가 4관점 평가:

| 관점 | 평가 기준 |
|------|----------|
| 👔 기획자 | 요구사항 충족도, 제약조건 반영, 엣지 케이스 |
| 💻 개발자 | 코드 품질, 에러 핸들링, 타입 안전성, 리소스 관리 |
| 🧪 테스터 | 테스트 커버리지, 한계 테스트, 에러 경로 |
| 👤 사용자 | 직관적 UI, 진행 피드백, 에러 메시지, 반응 속도 |

**4개 관점 모두 개별 90점 이상 → 다음 단계**
**하나라도 90 미만 → 보강 → 재평가 (최대 3회, 초과 시 이전 단계 재검토)**

1. **단계 건너뛰기 금지**: 이전 단계 산출물 필수
2. **평가 게이트 필수**: 모든 관점 개별 90+ 필수
3. **자체 순환**: 사용자 확인 불필요, 팀장끼리 교차 점검. 사용자는 진행 보고만 수신
4. **팀장 오케스트레이션**: 팀장이 팀원 작업 분배, 통합, 품질 관리
5. **병렬 극대화**: 독립 작업은 팀원 N명 동시 투입
6. **대기 없이 연속 진행**: 완료 즉시 다음 단계 시작
7. **서버 자동 재시작**: 백엔드/프론트엔드 수정 후 해당 서버 직접 재시작
8. **기획의도 수호**: 모든 테스트 결과는 기획팀장의 의도 판정을 거침
9. **수정→재테스트 순환**: 수정 시 반드시 QA팀장에게 재테스트 요청
10. **반복 제한**: 재평가 최대 3회, 수정→재테스트 최대 3회
11. **공통 모듈 선행**: 공통 개발자가 표준/공통 모듈을 먼저 만든 후 다른 개발자 투입
12. **표준 정보 필수 전달**: 개발자에게 작업 지시 시 코딩 표준 + 공통 모듈 문서 필수 첨부
13. **중복 구현 금지**: 공통 모듈에 이미 있는 기능을 개별 구현하면 리뷰에서 반려
14. **코드 크기 제한**: 리뷰어가 상한 초과 코드 발견 시 개발자에게 분할 지시 (강제)
15. **실제 동작 검증 필수**: 코드 변경 후 빌드만이 아닌 실제 서버 기동 + API 호출로 동작 확인 (빌드 성공 ≠ 동작 성공)
16. **미사용 코드 감지 필수**: 파일 변경/삭제/생성 시 import 참조를 grep으로 확인. 사용처 없는 코드는 삭제
17. **보안 수정 → 공격 시뮬레이션**: 보안 취약점 수정 후 반드시 수정 전 공격 벡터로 재시도하여 차단 확인
18. **프론트/백엔드 구분 확인**: 코드 변경 시 해당 기능이 프론트엔드에서 실행되는지 백엔드에서 실행되는지 정확히 확인 후 작업
19. **Playwright 동작 검증 필수**: 웹 UI 코드 변경 시 Playwright로 실제 브라우저 동작 검증 필수. 빌드만 확인하고 넘어가는 것 금지. Playwright 미실행 = Phase 전환 차단
20. **연계 기능 테스트 필수**: 변경 기능뿐 아니라 데이터가 전달되는 모든 연계 기능(탭 간 전달, 저장/불러오기 등)도 Playwright로 검증
21. **평가 점수 = 동작 증거 기반**: Playwright 테스트 결과(스크린샷 + 에러 로그)가 없으면 평가 점수 산정 불가. 문서만으로 90+점 부여 금지
22. **"완료"의 정의 = 사용자가 실제로 쓸 수 있는 상태**: 코드가 있고 빌드가 되고 문서가 있어도, 사용자가 브라우저에서 기능을 사용할 수 없으면 미완료
23. **기능 의존성 자동 분석 필수 (Phase 0.5)**: 기능 요청 시 코딩 전에 반드시 입력 데이터 물리경로, URL↔경로 변환, 출력 저장 위치, 프론트 state 반영, 프로젝트 저장/복원, 에러 시나리오, 연계 기능을 전부 도출. "버튼만 만들고 뒤를 빠뜨리면" Phase 전환 차단. 사용자가 하나씩 지적해야 하는 것은 프로세스 실패
24. **물리적 파일 경로 추적 필수**: API/기능에서 파일을 다루면 반드시 "이 파일은 디스크 어디에 있는가?"를 먼저 확인. 서빙 URL(/tts/download/...)과 로컬 경로(cache/tts/...)의 변환 로직을 반드시 포함
25. **저장/복원 무결성**: 새로운 상태를 추가하면 반드시 getState(), setState(), restoreState, ProjectState 타입, useProjectManager 저장/복원 로직 전부에 반영. 하나라도 빠지면 데이터 소실
26. **기존 코드 검색 후 재사용 필수**: 코딩 전 `grep -r`로 관련 함수/유틸/훅/API를 반드시 검색. 기존에 있으면 재사용하거나 확장. 검색 없이 새로 만들면 "중복 구현"으로 리뷰 반려. 같은 로직이 2곳 이상이면 즉시 공통 유틸로 추출
27. **기능맵(function_map.md) 유지관리**: `docs/dev/function_map.md`는 프로젝트의 지도. 모든 기능 구현 전에 맵을 먼저 참조하고, 구현 후에 맵을 갱신. 맵에 등록 안 된 함수/API를 사용하면 경고. 새 유틸/훅/API 추가 시 맵 갱신 누락은 리뷰에서 반려

## 보안 규칙 (Security First — 전 Phase 공통)

모든 Phase에서 보안은 기능보다 우선합니다. 보안 위반은 P0로 즉시 수정합니다.

### 1. API 키 / 시크릿 관리

```
[절대 금지]
- NEXT_PUBLIC_ 접두사로 시크릿 키 노출 (브라우저 번들에 포함됨)
- 소스코드에 API 키, 비밀번호, 토큰 하드코딩
- .env 파일을 Git에 커밋
- 로그/에러 메시지에 시크릿 포함

[필수 패턴]
- 외부 API 키 → 서버 사이드 프록시 (Next.js API Routes, Backend)
- 환경변수 → .env.local (Git 제외), .env.example (키 값 없이 구조만)
- 클라이언트에서 필요한 키 → 서버 프록시를 통해 간접 호출
- YouTube API 키 등 클라이언트 필수 키 → localStorage에 사용자 직접 입력

[검증 방법]
- Phase 4 완료 시: `grep -r "NEXT_PUBLIC_.*KEY\|NEXT_PUBLIC_.*SECRET\|NEXT_PUBLIC_.*TOKEN" src/`
- 빌드 후: `.next/static/` 내 번들에서 API 키 문자열 검색
- Phase 5 보안 테스트: 브라우저 DevTools Network 탭에서 키 미노출 확인
```

### 2. 입력 검증 / 인젝션 방지

```
[프론트엔드]
- 사용자 입력을 innerHTML/dangerouslySetInnerHTML에 직접 삽입 금지
- URL 파라미터를 검증 없이 사용 금지
- eval(), new Function() 사용 금지
- React의 기본 XSS 방어 활용 (JSX 자동 이스케이프)

[백엔드]
- SQL 쿼리에 f-string/format 사용 금지 → ORM 또는 파라미터 바인딩
- 파일 경로에 사용자 입력 직접 사용 금지 → 경로 순회(../) 방지
- 커맨드 인젝션 방지: subprocess에 shell=True + 사용자 입력 금지
- JSON 파싱 시 스키마 검증 (Pydantic 등)

[검증 방법]
- Phase 4 리뷰어: innerHTML, eval, f-string SQL, shell=True 패턴 검색
- Phase 5 테스터: 악성 입력 테스트 (`<script>alert(1)</script>`, `'; DROP TABLE`, `../../etc/passwd`)
```

### 3. 인증 / 권한

```
[필수 패턴]
- 인증이 필요한 API → 미들웨어/데코레이터로 일괄 적용 (개별 엔드포인트 누락 방지)
- JWT/세션 토큰 → HttpOnly, Secure, SameSite=Strict 쿠키
- 비밀번호 저장 → bcrypt/argon2 해시 (평문 저장 절대 금지)
- 권한 체크 → 리소스 접근 전 소유자/역할 확인 (IDOR 방지)

[현재 프로젝트 해당 사항]
- 인증 미구현 시에도 향후 확장 고려하여 API 엔드포인트 설계
- 파일 업로드/다운로드 경로에 사용자 입력 검증
```

### 4. CORS / 네트워크 보안

```
[필수 패턴]
- CORS 허용 origin → 명시적 도메인 (와일드카드 * 프로덕션 금지)
- 개발: localhost 허용 / 프로덕션: 실제 도메인만 허용
- 외부 API 호출 시 timeout 설정 (무한 대기 방지)
- HTTPS 강제 (프로덕션)
- Rate Limiting 적용 (API 남용 방지)

[검증 방법]
- Phase 3 개발팀장: CORS 설정 검토
- Phase 5 테스터: 다른 origin에서 API 호출 테스트
```

### 5. 의존성 보안

```
[필수 패턴]
- 정기적 취약점 검사: `pnpm audit` / `pip audit`
- High/Critical 취약점 → P0로 즉시 업데이트
- 미사용 의존성 제거 (공격 표면 축소)
- lock 파일(pnpm-lock.yaml, uv.lock) Git 포함 (재현 가능한 빌드)

[검증 방법]
- Phase 4 완료 시: `pnpm audit --audit-level=high`
- Phase 5: 의존성 목록 검토, CDN 외부 스크립트 제거 확인
```

### 6. 파일 업로드 / 다운로드 보안

```
[필수 패턴]
- 업로드 파일 타입 검증 (확장자 + MIME + Magic bytes)
- 업로드 크기 제한
- 업로드 파일명 새로 생성 (사용자 제공 파일명 사용 금지 → 경로 순회)
- 저장 경로를 웹 루트 밖으로
- 다운로드 시 Content-Disposition: attachment

[현재 프로젝트 해당 사항]
- Whisper 오디오 업로드: 파일 크기/형식 검증
- 캐릭터 시트 이미지: 업로드 경로 검증
- 프로젝트 JSON: 스키마 검증 후 저장
```

### 7. 에러 처리 / 정보 노출 방지

```
[절대 금지]
- 스택 트레이스를 사용자에게 노출 (프로덕션)
- DB 쿼리/스키마 정보 에러 메시지에 포함
- 내부 파일 경로 노출
- 빈 catch 블록 (에러 삼킴)

[필수 패턴]
- 사용자: 한국어 친화적 에러 메시지
- 개발자: 서버 로그에 상세 에러 기록
- 프로덕션: Error Boundary로 앱 크래시 방지
- 에러 응답: 일관된 형식 { success: false, error: "메시지" }
```

### 8. 보안 체크리스트 (Phase별)

| Phase | 보안 점검 항목 |
|-------|--------------|
| Phase 1 (기획) | 민감 데이터 흐름 식별, 인증/권한 요구사항 정의 |
| Phase 3 (개발준비) | API 키 관리 방식 설계, CORS 정책 설계, 에러 처리 전략 |
| Phase 4 (구현) | 코드 리뷰 시 OWASP Top 10 체크, `pnpm audit`, API 키 미노출 확인 |
| Phase 5 (테스트) | 보안 테스트 전용 단계(Phase 5d), 악성 입력/경로 순회/XSS/인젝션 테스트 |
| Phase 7 (최종) | 번들 내 시크릿 검색, 의존성 취약점 0 High 확인, CORS 최종 확인 |

### 9. 보안 위반 시 대응

```
보안 이슈 발견 → 즉시 P0 이슈 등록 → 현재 작업 중단

심각도 분류:
- P0-Critical: API 키 노출, SQL 인젝션, 인증 우회 → 즉시 수정, 다른 작업 모두 중단
- P0-High: XSS, CSRF, 경로 순회, 빈 catch → 현재 Phase 내 수정
- P1: 의존성 취약점, CORS 설정 → 다음 Phase 전 수정
- P2: 정보 노출(스택트레이스 등) → 예정된 수정

대응 흐름:
QA팀 발견 → QA팀장 → 기획팀장(판정) → 개발팀장(수정) → QA팀장(재테스트)
개발팀 발견 → 개발팀장(즉시 수정) → QA팀장(확인 테스트)
```
