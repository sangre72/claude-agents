# 표준 개발 프로세스 스킬

사용자가 `/dev` 명령어를 실행하면 **표준 개발 프로세스**를 자동으로 순환 실행합니다.

## 사용법

```bash
/dev 이미지→영상 변환 기능 추가        # 전체 프로세스 실행
/dev research 영상 생성 메모리 한계     # 조사만 실행
/dev design 씬 청크 분할 시스템         # 설계만 실행
/dev implement 이미지→영상 API          # 개발만 실행
/dev test 영상 생성 API                 # 테스트만 실행
/dev ux-test 시나리오 영상 생성         # UX 테스트만 (Playwright + 자동수정)
/dev review 시나리오 영상 생성          # 리뷰만 실행
/dev status                             # 전체 진행 현황 (DASHBOARD.md)
/dev status 영상 생성                   # 특정 기능 진행 현황
/dev checklist 영상 생성                # 세부 체크리스트 확인
```

## 실행 프로세스

사용자 요청을 받으면 아래 순서로 에이전트를 자동 호출합니다.

### 인자 파싱

- 인자 없음 또는 기능 설명만: 전체 프로세스 (research → design → implement → test → review)
- `research [주제]`: dev-researcher 에이전트만 실행
- `design [주제]`: dev-planner 에이전트만 실행
- `implement [주제]`: dev-developer 에이전트만 실행
- `test [주제]`: dev-tester 에이전트만 실행
- `ux-test [주제]`: dev-ux-tester 에이전트만 실행 (Playwright + 자동 수정)
- `review [주제]`: dev-reviewer 에이전트만 실행
- `status [기능명?]`: 진행 현황 대시보드 표시
- `checklist [기능명]`: 세부 체크리스트 확인/갱신

### 전체 프로세스 실행 순서

**Step 1: 기획 파악**
사용자 요청에서 요구사항, 제약조건, 기대 산출물을 정리합니다.
기획 결과는 dev-developer에게 전달하여 기술적 실현 가능성, 필요 모듈, 라이브러리를 조사하게 합니다.

**전 단계 공통: TRACKER (추적 관리자)**
```
dev-tracker 에이전트가 모든 단계에서 상시 실행됩니다:
- 코드베이스 스캔 → 변경/추가 파일의 함수/클래스/메서드 추출
- 세부 체크리스트 자동 생성 (함수 하나까지 추적)
- 각 단계 완료 시 체크 표시 갱신
- DASHBOARD.md 실시간 업데이트
- 이슈 발견 시 issues.md에 자동 기록
- 산출물: docs/tracker/DASHBOARD.md + 기능별 체크리스트
```

사용자는 언제든 `/dev status`로 현황을 확인할 수 있습니다.

**Step 2: RESEARCH (조사)**
```
dev-researcher 에이전트를 실행합니다:
- 기술 조사 (라이브러리, 모델, API)
- 리소스 측정 (메모리, 디스크, GPU)
- 한계 파악 (OOM 조건, 최대 해상도 등)
- 산출물: docs/research/[기능명]_research.md
```

결과를 dev-evaluator에게 평가 요청합니다. 4관점 모두 90+ 시 자동 진행.

**Step 3: DESIGN (설계)**
```
dev-planner 에이전트를 실행합니다:
- 조사 리포트 기반 아키텍처 설계
- 모듈 분리 및 인터페이스 정의
- 병렬 개발 계획
- 산출물: docs/design/[기능명]_design.md
```

설계서를 dev-evaluator + dev-developer에게 점검 요청. 4관점 모두 90+ 시 자동 진행.

**Step 4: IMPLEMENT (개발)**
```
dev-developer 에이전트를 실행합니다 (병렬 가능):
- 설계서 기반 코드 구현
- 스키마 → 백엔드 → 프론트엔드 순서
- 독립 모듈은 병렬 에이전트로 실행
- 빌드 확인
- 백엔드 코드 수정 시 서버 자동 재시작 (사용자에게 요청하지 않음)
  1. 기존 uvicorn 프로세스 kill
  2. 새로 실행
  3. 서버 ready 확인 후 다음 단계
```

**Step 5: TEST (테스트) - 11개 카테고리 필수**
```
dev-tester 에이전트를 실행합니다:
1. 스모크 테스트 (기본 동작)
2. 단위 테스트 (함수별)
3. 통합 테스트 (전체 흐름 E2E)
4. 한계 테스트 (OOM, 타임아웃, 대용량)
5. 보안 테스트 (API 키 노출, XSS, 경로 순회, 파일 타입)
6. 외부 리소스 (핫링크, CORS, 레이트리밋, 타임아웃)
7. 동시성 (중복 요청, 경쟁 조건, 상태 충돌)
8. 데이터 무결성 (타입 일치, null 처리, 인코딩)
9. 리소스 정리 (임시 파일, 메모리 해제, 타이머 정리)
10. 환경/배포 (환경변수 누락, 바이너리 미설치)
11. 에러 복구 (실패 후 재시도, 정상 복귀)
- 산출물: docs/test/[기능명]_test.md
```

테스트 실패 시:
- 에러 정보를 dev-developer에게 전달
- 수정 후 재테스트 (최대 3회)
- 3회 초과 시 설계 재검토 제안

**Step 6: UX TEST (사용자 관점 테스트 + 자동 수정)**
```
dev-ux-tester 에이전트를 실행합니다 (Playwright):
- 사용자 흐름 시나리오 자동 테스트
- 로딩/진행 상태 확인 (버튼 비활성화, 프로그레스바 등)
- 에러 처리 UX (에러 메시지, 재시도 가능 여부)
- 반응성 (클릭 후 200ms 이내 피드백)
- 접근성 (키보드, aria-label, 색상 대비)
- 스크린샷 자동 캡처
- 산출물: docs/ux-test/[기능명]_ux.md + screenshots/
```

문제 발견 시 자동 수정:
1. 원인 분석 (DOM, API, 타이밍 등)
2. 코드 수정 적용
3. 재테스트로 수정 검증
4. 최대 3회 반복, 수정 불가 시 리포트에 기록

**Step 7: REVIEW (최종 리뷰)**
```
dev-reviewer 에이전트를 실행합니다:
- 사용성 (UX) 점검
- 에러 처리 완성도
- 코드 품질
- 성능/리소스 효율
- 산출물: docs/review/[기능명]_review.md
```

리뷰 결과에 따라:
- 심각한 문제 → 설계 또는 개발로 돌아감
- 경미한 개선 → 개선 후 완료
- 양호 → 완료 보고

### 평가 게이트 (모든 단계 공통)

매 단계 완료 시 **dev-evaluator**가 4개 관점으로 평가합니다:

| 관점 | 평가 기준 |
|------|----------|
| 👔 기획자 | 요구사항 충족도, 제약조건 반영, 엣지 케이스 |
| 💻 개발자 | 코드 품질, 에러 핸들링, 타입 안전성, 리소스 관리 |
| 🧪 테스터 | 테스트 커버리지, 한계 테스트, 에러 경로 |
| 👤 사용자 | 직관적 UI, 진행 피드백, 에러 메시지, 반응 속도 |

**4개 관점 모두 개별 90점 이상 → 다음 단계 진행**
**하나라도 90점 미만 → 문제점 분석 → 보강 → 재평가**
(평균으로 상쇄 불가. 기획자 100, 개발자 100, 테스터 89, 사용자 100 → 불합격)

- 미달 관점의 원인을 먼저 파악한 후 보강
- 재평가 최대 3회. 초과 시 이전 단계 재검토 제안
- 점수 이력은 `docs/evaluation/[기능명]_[단계]_eval.md`에 보존

### 순환 규칙

1. **단계 건너뛰기 금지**: 각 단계는 이전 단계의 산출물이 있어야 진행
2. **평가 게이트 필수**: 모든 단계는 4관점 모두 개별 90점 이상이어야 다음 단계
3. **에이전트 자체 점검**: 사용자 확인 불필요. 에이전트끼리 교차 점검 + dev-evaluator 평가. 사용자는 진행 보고만 수신
4. **산출물 저장**: 모든 산출물은 `docs/` 디렉토리에 저장
5. **반복 제한**: 재평가 최대 3회, 초과 시 이전 단계 재검토
6. **대기 없이 연속 진행**: 에이전트 완료 즉시 다음 단계를 시작한다. 보고를 위해 멈추지 않는다. 병렬 가능한 단계는 동시 실행한다. 조사+설계, 설계+구현, 구현+테스트를 겹쳐서 진행할 수 있다.
7. **서버 자동 재시작**: 백엔드 코드 수정 후 서버 재시작을 직접 수행. 사용자에게 요청하지 않는다.

### 산출물 디렉토리

```
docs/
├── research/      # 조사 리포트
├── design/        # 설계서
├── test/          # 테스트 리포트
├── ux-test/       # UX 테스트 리포트 + 스크린샷
├── evaluation/    # 단계별 평가 리포트 (점수 이력)
├── review/        # 최종 리뷰 리포트
└── tracker/       # 체크리스트 + DASHBOARD.md
```

첫 실행 시 `docs/` 디렉토리가 없으면 생성합니다.
