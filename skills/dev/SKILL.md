# /dev — 개발 프로세스

기능 요청 → 크기 판단 → 분석 → 구현 → 검증. Hook이 자동 감시.

## 사용법

```bash
/dev [기능]              # 크기 자동 판단 → 적절한 경로 실행
/dev research [주제]     # 기술/사용자 리서치만
/dev test [기능]         # 테스트만
/dev status              # 진행 현황
```

---

## 1단계: 크기 판단 (자동)

요청을 받으면 먼저 크기를 판단한다. **사용자에게 묻지 않고 스스로 결정.**

| 크기 | 기준 | 예시 |
|------|------|------|
| **S** | 파일 1~3개 수정, 기존 패턴 반복 | 버그 수정, 텍스트 변경, 스타일 조정 |
| **M** | 파일 3~10개, 백엔드+프론트 연결 | 버튼 추가, API 연결, 새 모달 |
| **L** | 새 탭/페이지/시스템, 10개+ 파일 | 새 파이프라인, 새 기능 영역 |

---

## 2단계: 크기별 경로

### S — Fast (분석 → 구현 → Hook 검증)

```
1. grep으로 변경 대상 파악
2. 구현
3. 빌드 확인
4. Hook이 자동 검증 → 끝
```
문서: 없음. 커밋 메시지가 문서.

### M — Standard (분석 → 구현 → Playwright)

```
1. 의존성 분석 (E2E 체인 체크리스트)
2. 기존 코드 검색 (재사용 우선)
3. 구현 (백엔드 → 프론트 → 연결 → 테스트)
4. Playwright 동작 검증
5. Hook 자동 검증 → 끝
```
문서: 없음. 코드 + 테스트가 문서.

### L — Full (리서치 → 설계 → 구현 → 테스트)

```
1. 리서치 (WebSearch: 사용자 니즈, 경쟁사, 기술 비교)
2. 의존성 분석 + 설계 (데이터 플로우, API 설계)
3. 구현 (공통 모듈 선행 → 병렬 개발)
4. Playwright E2E 테스트
5. 보안 검토 (pnpm audit, API 키, 인젝션)
6. 통합 시나리오 테스트 → 끝
```
문서: 설계서 1개만 (`docs/dev/[기능명]_design.md`). 나머지는 코드가 문서.

---

## 의존성 분석 (M, L 공통) ★ 가장 중요

코딩 전에 반드시 실행. **이것만 제대로 하면 90% 버그 방지.**

```
□ 기존 코드 검색 (grep) — 재사용 가능한 것?
□ 백엔드 API → 프론트 호출 코드 필요?
□ API 응답 → React state 반영 경로?
□ state → UI 표시 컴포넌트?
□ 탭 전환 → 데이터도 함께 전달?
□ 새 state → getState/setState/ProjectState/useProjectManager 전부 반영?
□ 프로젝트 생성 → 로드 → 화면 표시까지 연결?
□ 컴포넌트 간 콜백/store 연결?
□ 파일 경로: 디스크 위치 ↔ 서빙 URL 변환?
□ 에러 시나리오: 빈 데이터, 네트워크 실패, 타임아웃?
```

**"버튼 클릭 → 화면에 결과 표시"까지 전체 체인이 없으면 구현 시작 금지.**

---

## 구현 규칙 (전 크기 공통)

1. **기존 코드 재사용**: grep 검색 → 있으면 재사용/확장. 새로 만들지 않음
2. **E2E 체인 완성**: API만/UI만/탭 전환만 = 미완료. 끝까지 연결
3. **막다른 에러 금지**: "먼저 X 해주세요" 에러 = UX 실패. 대신 자동으로 다음 단계를 안내/실행. 버튼 눌렀는데 에러만 나오면 안 됨 — 필요한 것을 바로 보여줘야 함
4. **서버 자동 재시작**: 코드 수정 후 해당 서버 직접 재시작
5. **영속 데이터 = DB**: 모듈 변수(dict/set) = 재시작 시 소실 → DB 저장
6. **보안**: API 키 노출/eval/인젝션 = 즉시 수정

## 검증 (Hook이 자동 실행 — 기억할 필요 없음)

| 시점 | Hook | 뭘 잡나 |
|------|------|--------|
| 요청 입력 | UserPromptSubmit | E2E 리마인드 자동 주입 |
| .tsx 수정 | PostToolUse | `?.` 누락, eval, XSS → **차단** |
| git commit | PreToolUse | 테스트 없이 커밋 → **차단** |
| 응답 완료 | Stop | API↔프론트 누락 → **기획 검증 + 재작업** |
| 컨텍스트 압축 | PostCompact | 핵심 규칙 리마인드 |
| 세션 시작 | SessionStart | `.claude/rules/` 존재 확인 |

**Claude가 규칙을 잊어도 Hook이 차단한다. 그래서 규칙을 많이 외울 필요 없음.**

## Playwright 테스트 (M, L만)

웹 UI 변경 시 필수. 빌드만 확인하고 넘어가면 안 됨.

```bash
npx playwright test tests/target-test.spec.ts --reporter=list
```

테스트 포함: page.on('pageerror'), page.on('requestfailed'), page.screenshot().
범위: ① 직접 기능 ② 연계 기능(탭 간 데이터) ③ 에러 시나리오.

---

## Sub-agent 투입 기준

| 크기 | 투입 | 방식 |
|------|------|------|
| S | 없음 | 직접 구현 |
| M | 필요 시 1~2개 | 독립 작업 병렬 (프론트/백엔드) |
| L | 탐색 + 구현 병렬 | Explore agent → 구현 agent 다수 |

**L에서만 "팀" 개념 적용. S/M에 팀 오버헤드 금지.**

## 하지 않는 것

```
❌ 매번 docs/ 산출물 9종 생성 (코드가 문서)
❌ 자기가 자기를 98점 평가 (의미 없음)
❌ 7명 전문가 교차 검증 (S/M에 과도)
❌ Phase 9단계 순차 실행 (크기에 맞게 축소)
❌ "계속할까요?" 질문 (자동 진행)
❌ 사용자 지시 없이 리서치 시작 (S/M은 바로 구현)
```

## 완료의 정의

**사용자가 브라우저에서 기능을 쓸 수 있으면 완료.**
코드가 있어도, 빌드가 돼도, 문서가 있어도 — 사용자가 쓸 수 없으면 미완료.
