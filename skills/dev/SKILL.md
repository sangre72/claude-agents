# 표준 개발 프로세스 스킬 (팀 조직 체계)

`/dev` 명령어 → 4개 팀(기획/디자인/개발/QA) 팀장-팀원 체계로 자동 순환 실행.

---

## 최우선 원칙

**빌드 성공 ≠ 동작 성공. Playwright PASS 없이 어떤 Phase도 완료 불가.**

모든 Phase 전환 시 Playwright로 검증: 페이지 로드, 핵심 기능 동작, 콘솔 에러 0건, API 성공, 스크린샷 증거.
테스트 시나리오: ① 직접 기능 ② 연계 기능(탭 간 데이터 전달) ③ 사용자 시나리오 E2E ④ 에러 시나리오.

## 사용법

```bash
/dev [기능]              # 전체 프로세스
/dev planning [기능]     # 기획팀 (Phase 0+1)
/dev design [기능]       # 디자인팀 (Phase 2)
/dev research [주제]     # 기술 리서처
/dev implement [기능]    # 개발팀 (Phase 4)
/dev test [기능]         # QA팀 (Phase 5)
/dev ux-test [기능]      # UX테스터
/dev review [기능]       # 코드 리뷰
/dev scenario-test [기능] # 통합 시나리오 (Phase 8)
/dev status              # 진행 현황
```

---

## 팀 조직

```
dev-orchestrator (총지휘)
├── 기획팀장 → 사용자리서처, 기술리서처, 설계자, 평가자
├── 디자인팀장 → UI/UX/시스템 팀원
├── 개발팀장 → 공통개발자(선행), 개발자N, 리뷰어
└── QA팀장 → 테스터, UX테스터, 보안테스터
```

각 분야 최소 7명, **찬성 2 + 반대 1** 교차 검증. 반론자 지적 타당하면 반드시 반영.

---

## Phase 실행 순서

### Phase -1: 이전 작업 확인
- `docs/tracker/DASHBOARD.md` 확인 → 미완료 있으면 이어서 진행
- `.gitignore` dev process 패턴 확인/추가
- `docs/` 하위 디렉토리 구조 확인/생성

### Phase 0: 리서치
- 사용자 리서처: WebSearch로 사용자 니즈, 경쟁사, UX 패턴 조사
- 기술 리서처: 최신 안정 기술 비교, **패키지 CVE/보안 취약점 조사**, `pnpm audit`
- 기획팀장: 리서치 종합 → 기능 스코프 결정

### Phase 0.5: 의존성 그래프 ★★★

**기능 요청 → 코딩 전에 반드시 전체 데이터 플로우 분석. 누락 0건이어야 기획 시작.**

```
0.5-0. 기존 코드 검색 (grep) — 재사용 우선, 중복 구현 금지
0.5-1. 입력 데이터 물리 경로 추적 (디스크 위치, URL↔경로 변환)
0.5-2. 처리 파이프라인 (도구, 비동기 여부, 에러 시나리오)
0.5-3. 출력/저장 (파일 위치, 이름, project.json 반영)
0.5-4. 프론트 state 연계 (어떤 컴포넌트가 결과 표시, props 체인)
0.5-5. 저장/복원 (getState/setState/ProjectState/useProjectManager 전부 반영)
0.5-6. API 설계 (엔드포인트, 스키마, 보안)
0.5-7. E2E 체인 검증 체크리스트:
       □ 백엔드 API → 프론트 호출 코드?
       □ API 응답 → React state 반영?
       □ state → UI 표시 컴포넌트?
       □ 탭 전환 → 데이터도 함께 전달?
       □ 프로젝트 생성 → 로드 → 화면 표시?
       □ 컴포넌트 간 콜백/store 연결?
```

### Phase 1: 기획
- 기존 코드 현황 분석 (변경 대상 사용처 grep, 실행 환경 확인)
- 요구사항서 작성 → 설계 지시 → 통합 + 검증 → 4관점 평가 98+

### Phase 2: 디자인
- 기획의도 파악 → 팀원 분배 → 통합 → 기획팀장과 의도 확인 회의 → 평가 98+

### Phase 3: 개발 준비
- 플로우 설계 → 공통/모듈화 분석 → 기획팀장 논리 검증 회의 → 기술 검토 → 평가 98+

### Phase 4: 구현

```
4-0. 공통 개발자: 표준화 + 공통 모듈 선행 (다른 개발자보다 먼저)
4-1. 작업 분배 (코딩 표준 + 공통 모듈 문서 필수 첨부)
4-2. 코드 구현 (병렬, 기존 모듈 재사용 우선)
     - 코딩 전: 기능 위치 확인, 영향 범위 grep, 프론트/백엔드 구분
4-3. 코드 리뷰 (표준 준수, 공통 모듈 활용, 크기 제한: 컴포넌트 500줄/훅 300줄/유틸 200줄)
4-4. 통합 + 빌드 + 서버 재시작
4-5. ★ Playwright 동작 검증 (필수 — 이것 없이 Phase 전환 불가)
     - 실제 버튼 클릭/입력, 콘솔 에러 0건, 연계 기능(탭 간 전달, 저장/불러오기), 스크린샷
4-6. 4관점 평가 98+ (Playwright 결과 필수)
```

### Phase 5: 테스트
- 전략 수립 → 스모크(실제 서버) → 단위 → 통합 E2E → **보안** → 한계 → **UX Playwright** → 리포트 → 평가 98+

### Phase 6: 기획 검토
- QA 결과 → 기획팀장 판정 → OK면 Phase 7 / MISMATCH면 수정→재테스트 순환

### Phase 7: 최종 리뷰
- 전 팀장 합동 점검 → 최종 평가 98+ → 승인서

### Phase 8: 통합 시나리오 테스트 ★★★ 최종 게이트
- 기획자 시나리오 기반 E2E → Playwright로 전 단계 재현 → 전체 PASS → 개발 완료

## 평가 게이트

매 Phase: 4관점(기획/개발/테스트/사용자) 각각 **98+**. 미달 시 보강→재평가 자동 진행.

---

## 핵심 규칙 (12개)

1. **Phase 생략 금지**: 0→1→2→3→4→5→6→7→8 전부 실행. 자의적 중단 = 프로세스 위반
2. **자동 진행**: 98점 미달 시 사용자 확인 없이 즉시 보강→재평가. "계속할까요?" 금지
3. **E2E 체인 완성 필수**: 모든 기능은 "버튼 클릭 → 화면에 결과 표시"까지 전체 체인. API만, UI만, 탭 전환만은 미완료
4. **기존 코드 재사용**: 코딩 전 grep 검색 필수. 있으면 재사용/확장. 중복 구현 = 리뷰 반려
5. **저장/복원 무결성**: 새 state 추가 시 getState/setState/restoreState/ProjectState/useProjectManager 전부 반영
6. **영속 데이터 = DB**: 서버 재시작 후 유지 필요 → DB 저장 필수. 모듈 변수(dict/set) = 재시작 시 소실
7. **서버 자동 재시작**: 코드 수정 후 해당 서버 직접 재시작
8. **Playwright 필수**: 웹 UI 변경 시 실제 브라우저 동작 검증 + 연계 기능 + 스크린샷. 미실행 = Phase 전환 차단
9. **공통 모듈 선행**: 공통 개발자가 표준/모듈을 먼저 만든 후 다른 개발자 투입
10. **수정→재테스트 순환**: 수정 후 반드시 QA에 재테스트 요청. 스킵 불가
11. **기능맵 유지**: `docs/dev/function_map.md` — 구현 전 참조, 구현 후 갱신
12. **보안 우선**: API 키 노출/인젝션/XSS = P0 즉시 수정. `pnpm audit` High 0건

## 보안 체크리스트 (전 Phase 공통)

| 항목 | 금지 | 필수 |
|------|------|------|
| API 키 | NEXT_PUBLIC_ 시크릿, 하드코딩, .env 커밋 | 서버 프록시, .env.local |
| 입력 | innerHTML, eval, f-string SQL, shell=True | React JSX 이스케이프, Pydantic, 파라미터 바인딩 |
| 파일 | 사용자 파일명 직접 사용, 경로 순회 | 파일명 재생성, resolve()+startswith() |
| 에러 | 스택트레이스 노출, 빈 catch | { success: false, error: "메시지" } |
| 의존성 | High/Critical 취약점 | `pnpm audit` / `pip audit`, lock 파일 Git 포함 |
